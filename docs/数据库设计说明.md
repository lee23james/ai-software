# 数据库设计说明

## 1. 设计目标

本数据库设计服务于“面向大学生就业的招聘数据分析与智能求职辅助平台”，支撑前端岗位列表、岗位详情、数据分析、简历辅助和后台管理等模块。

当前项目后端默认使用 H2 内存数据库，并预留 MySQL 接入能力。因此表结构设计以 MySQL 为主要目标，同时尽量避免过度依赖特定数据库能力，方便在 H2 的 MySQL Mode 下做本地开发和演示。

核心目标如下：

- 支撑岗位数据的导入、查询、筛选和详情展示。
- 支撑学生账号、求职档案、简历上传和简历解析。
- 支撑岗位收藏、岗位投递、投递状态跟踪。
- 支撑 AI 简历解析、技能提取、岗位匹配和优化建议结果落库。
- 支撑城市岗位分布、热门技能、平均薪资等数据分析指标。
- 支撑后台管理中的数据导入、数据清洗和操作审计。

## 2. 设计原则

### 2.1 业务表优先，避免过早复杂化

当前项目是课程项目基础框架，不适合一开始就引入复杂的数据仓库模型。数据库先围绕核心业务对象建模，包括用户、学生档案、公司、岗位、简历、技能、收藏、投递和 AI 匹配记录。

### 2.2 保留可分析字段

岗位数据不仅用于列表展示，也用于城市分布、薪资区间、技能热度等统计分析。因此岗位表中保留城市、薪资下限、薪资上限、学历、经验、发布时间等结构化字段。

### 2.3 技能既保留原始标签，也拆成关联表

`job_info.skill_tags` 保留原始技能字符串，方便兼容当前接口和示例 CSV 数据；同时通过 `skill_dict`、`job_skill`、`resume_skill` 进行结构化建模，方便后续做技能热度分析、简历技能画像和岗位匹配。

### 2.4 AI 结果可追溯

简历解析、岗位匹配和简历建议不只返回给前端，也应保存到数据库。这样可以减少重复调用 AI 服务，也方便后台统计用户使用情况和排查匹配结果。

### 2.5 字段命名统一

表名和字段名统一使用小写下划线风格，例如 `job_info`、`salary_min`、`created_at`。Java 实体类通过 MyBatis-Plus 的驼峰映射能力转换为 `jobInfo`、`salaryMin`、`createdAt`。

## 3. 表结构总览

| 表名 | 中文含义 | 主要用途 |
| --- | --- | --- |
| `sys_user` | 用户账号表 | 登录、权限区分、用户基础信息 |
| `student_profile` | 学生档案表 | 学校、专业、年级、求职意向 |

| `resume_parse_result` | 简历解析结果表 | AI 解析后的结构化结果和建议 |
| `resume_skill` | 简历技能关联表 | 学生技能画像、匹配计算 |

## 4. 核心关系说明

```text
sys_user 1 ── 0..1 student_profile
sys_user 1 ── 0..N resume_info
sys_user 1 ── 0..N job_favorite
sys_user 1 ── 0..N job_application

company_info 1 ── 0..N job_info

job_info N ── N skill_dict，通过 job_skill 关联
resume_info N ── N skill_dict，通过 resume_skill 关联

resume_info 1 ── 0..N resume_parse_result
resume_info 1 ── 0..N job_match_record
job_info 1 ── 0..N job_match_record

data_import_batch 1 ── 0..N job_info
```

说明：

- 一个用户可以是学生，也可以是管理员，通过 `sys_user.role` 区分。
- 学生用户可以维护一份求职档案，也可以上传多份简历。
- 一个公司可以发布多个岗位。
- 岗位和技能是多对多关系。
- 简历和技能是多对多关系。
- 一次 AI 匹配记录同时关联一份简历和一个岗位。
- 数据导入批次用于追踪岗位数据来源和导入质量。

## 6. 和现有功能的对应关系

### 6.1 首页看板

首页当前展示：

- 岗位总数。
- 覆盖城市。
- 热门技能。
- 平均薪资。

对应数据来源：

- `job_info`：实时计算岗位总数、覆盖城市、平均薪资。
- `job_skill`、`skill_dict`：统计热门技能。
- `analysis_metric_snapshot`：缓存上述指标。

### 6.2 岗位列表和岗位详情

对应数据来源：

- `job_info`：岗位名称、公司、城市、薪资、学历、经验、描述。
- `company_info`：公司详细信息。
- `job_skill`、`skill_dict`：岗位技能要求。

查询条件建议：

- 城市：`job_info.city`。
- 关键词：匹配 `job_info.job_name`、`job_info.company_name`、`job_info.skill_tags`。
- 薪资区间：匹配 `salary_min`、`salary_max`。
- 技能：通过 `job_skill.skill_id` 查询。

### 6.3 数据分析

对应数据来源：

- 城市岗位分布：按 `job_info.city` 聚合。
- 平均薪资：按 `(salary_min + salary_max) / 2` 聚合。
- 热门技能：按 `job_skill.skill_id` 聚合。
- 指标缓存：保存到 `analysis_metric_snapshot`。

### 6.4 简历辅助

对应数据来源：

- `resume_info`：保存上传文件、简历文本和目标岗位。
- `resume_parse_result`：保存 AI 解析结构。
- `resume_skill`：保存简历技能。
- `job_match_record`：保存岗位匹配分数和建议。

### 6.5 后台管理

对应数据来源：

- `job_info`、`company_info`：岗位和公司管理。
- `sys_user`、`student_profile`：用户管理。
- `data_import_batch`：岗位数据导入管理。
- `operation_log`：后台操作审计。

## 10. 实施顺序建议

建议按以下顺序落地数据库能力：

1. 建立 `company_info`、`job_info`、`skill_dict`、`job_skill`，先让岗位列表、岗位详情和城市统计从数据库查询。
2. 建立 `sys_user`、`student_profile`，让登录和用户管理具备真实数据基础。
3. 建立 `resume_info`、`resume_parse_result`、`resume_skill`，接入简历上传和 AI 简历解析。
4. 建立 `job_match_record`，保存岗位匹配结果和建议。
5. 建立 `job_favorite`、`job_application`，完善学生求职流程。
6. 建立 `analysis_metric_snapshot`，优化首页和分析页统计性能。
7. 建立 `data_import_batch`、`operation_log`，完善后台数据导入和审计能力。

## 11. 注意事项

- 后端当前默认使用 H2 内存数据库，数据会随服务重启丢失。正式开发真实数据时建议切换到 MySQL。
- AI 返回的 JSON 结果建议先用 `TEXT` 或 `LONGTEXT` 保存，避免 H2 和 MySQL JSON 类型差异影响本地调试。
- 密码必须保存哈希值，不能保存明文。
- `job_info.company_name` 虽然和 `company_info.company_name` 有冗余，但可以降低 CSV 导入和早期接口开发成本。
- 后续如果岗位数据量明显增长，可以再考虑把分析相关数据拆成更标准的事实表和维度表。
